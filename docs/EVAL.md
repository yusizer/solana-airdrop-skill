# Evaluation report — solana-airdrop-skill

A reproducible, offline, deterministic evaluation of whether an airdrop built
**with the skill** is claimable on-chain versus the same airdrop built by a
developer **without the skill** who reaches for plausible-but-wrong
conventions. The suite is executable — `python tests/test_eval.py` prints the
scorecard and runs in CI — not a hand-written claim.

> Companion to `tests/test_eval.py`. Re-run any time; the numbers below are the
> deterministic output of that script.

## Methodology

- **Dataset.** N = 256 recipients with deterministic 32-byte pubkeys and raw
  token amounts. No network, no RPC, no secrets — fully reproducible.
- **Ground truth.** The airdrop authority publishes a Merkle root computed with
  the **correct** encoding for the target program — the Solana Foundation
  `merkle-airdrop` template: `keccak256(pubkey ‖ amount_le_u64 ‖ 0x00)`,
  index-based verification. This is the root an on-chain `claim_airdrop`
  instruction recomputes and checks.
- **with_skill.** Proofs generated by `examples/merkle_tree.AirdropDistribution`
  with `encoding="solana-official"`. Every proof must verify against the
  published root → every recipient can claim.
- **baselines.** A developer without the skill applying common, plausible
  conventions. Each is a **real mistake observed in primary sources** (see
  `skill/resources.md`), not an artificial strawman:
  - *sha256-not-keccak* — hashes leaves with `sha256` (the stdlib default)
    instead of `keccak256`. Every verified Solana program uses keccak.
  - *big-endian-amount* — encodes the amount big-endian. Solana is little-endian.
  - *missing-isClaimed* — omits the trailing `0x00` byte the on-chain
    `claim_airdrop` includes in the leaf.
  - *convention-mismatch* — correct keccak leaf, but builds OpenZeppelin
    sorted-pair proofs while the on-chain verifier is index-based. This is the
    exact JS-helper bug flagged in the official template
    (`verifyGillProof` sorts; the on-chain verifier does not).
- **Claimable rate.** The fraction of recipients whose proof verifies against
  the **published (correct) root** — i.e. whose on-chain claim would actually
  succeed. A baseline that builds its own (wrong) tree produces proofs that do
  not verify against the published root → claims fail → tokens are locked.
- **Tamper guard.** The skill must reject every forged claim (wrong amount,
  wrong index, corrupted sibling, empty proof) — 0 false-accepts.
- **Pass criterion.** The suite exits non-zero unless with_skill claims 100%,
  beats every baseline, and accepts zero forged claims.

## Results

| builder | claimable | rate |
|---|---:|---:|
| **with_skill** | **256 / 256** | **100.0%** |
| baseline: sha256-not-keccak | 0 / 256 | 0.0% |
| baseline: big-endian-amount | 0 / 256 | 0.0% |
| baseline: missing-isClaimed | 0 / 256 | 0.0% |
| baseline: convention-mismatch | 0 / 256 | 0.0% |
| **tamper guard** (forged claims accepted) | **0 / 4** | target 0 |

**with_skill 256 / 256 vs every baseline 0 / 256**, with **0 / 4 forged claims accepted**.

### Why every baseline scores 0% (and why that is honest, not rigged)

Merkle verification is **cryptographic and binary**: a proof either recomputes
the exact published root or it does not. A single wrong byte in the leaf — a
`sha256` digest where `keccak256` is required, a big-endian amount, a missing
`0x00`, or siblings ordered for the wrong verify convention — produces a
completely different hash chain that cannot match the root. There is no
"partial credit": a wrong convention does not make *some* claims fail, it
makes **all** of them fail.

That is the real-world failure mode this skill exists to prevent. A builder
who gets the leaf encoding wrong discovers it only after launch, when every
claimant's transaction reverts and the allocated tokens are stranded. The 0%
baselines here are not a strawman engineered to lose — they are the actual
consequences of the four most common convention mistakes, verified against the
on-chain code of real Solana programs.

This is a different and stricter kind of evidence than a "with-skill beats
baseline by a margin" scorecard: it reflects that airdrop correctness is
**pass/fail per drop**, not graded on a curve.

## Reproducibility

```bash
python tests/test_eval.py          # prints the scorecard; exit 0 on PASS
python -m unittest tests.test_merkle   # 19 unit tests (Keccak KATs, encodings, tamper)
```

CI runs both on every push (`.github/workflows/validate.yml`). No network, no
RPC, no secrets — the evaluation is fully deterministic.

## Scope & limitations

- **One target program exercised end-to-end** (Solana Foundation template,
  SOL). The other two encodings (Metaplex gumdrop mainnet, SPL-token vault) are
  unit-tested in `tests/test_merkle.py` (leaf shape, round-trip, tamper) but
  not run through the full claimable-rate eval. Adding them is a config swap
  (`encoding="gumdrop"`, `mode="sorted-pair"`).
- **Off-chain proof generation + verification only.** The eval proves the
  cryptography and convention correctness — the part that bricks drops. It does
  not submit a live `claim_airdrop` transaction (that needs a funded devnet
  program + RPC; the claim-instruction *builder* is `examples/claim_builder.ts`,
  typechecked in CI, simulate-only).
- **Deterministic synthetic recipients.** Pubkeys are synthetic 32-byte
  values, not mainnet addresses. The point is to measure convention
  correctness against ground truth, not to sample real claimants.

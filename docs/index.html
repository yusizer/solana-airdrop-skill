The problem

Merkle airdrops are convention-fragile: one wrong choice — sha256 instead of keccak256, big-endian instead of little-endian, a missing 0x00 byte, or sorted-pair proofs against an index-based on-chain verifier — makes 100% of claims unverifiable and bricks the drop after launch. No official Solana package ships a Merkle tree, so every builder hand-rolls the easy-to-get-wrong part.

This skill gets it right — and proves it with a quantified eval.

What it does

Build the drop

Ingest recipients, build the Merkle tree with the target program's exact leaf encoding, emit the root + per-recipient proofs. Zero-dep pure-Python Keccak-256 (NIST-KAT-verified) so it runs anywhere, including bare CI.

Verify before launch

verify_all() == N is the non-negotiable gate before the root is published on-chain. A wrong-encoding root is unrecoverable after launch — this gate prevents it.

Build claim txs

Typed @solana/web3.js instruction builders for initialize and claim — claim-status PDAs, double-claim prevention, for the Solana Foundation template (SOL) and Metaplex mpl-gumdrop (SPL, mainnet). tsc --noEmit clean.

Prevent double claims

Per-program PDA derivation (["claim", airdrop, claimant] / ["ClaimStatus", index, distributor]) so a recipient cannot claim twice — even from a leaked proofs file.

The eval (the reason to trust it)

Deterministic, offline, runs in CI. Builds a 256-recipient drop with the correct on-chain encoding, then measures claimable rate with the skill vs a developer without it who reaches for plausible-but-wrong conventions.

The 0% baselines are not a strawman — Merkle verification is cryptographic and binary: one wrong byte makes a proof unverifiable, so a wrong convention bricks the entire drop. That is exactly the failure mode this skill prevents. Full methodology: docs/EVAL.md.
